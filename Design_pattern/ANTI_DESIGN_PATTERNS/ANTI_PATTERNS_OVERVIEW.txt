                    ANTI-DESIGN PATTERNS OVERVIEW
================================================================================

WHAT ARE ANTI-PATTERNS?
================================================================================
Anti-patterns are COMMONLY-USED but INEFFECTIVE solutions to common problems
that result in poor design, maintainability issues, or performance problems.

They are:
- Recognizable patterns that appear to be good
- But actually cause more problems than they solve
- Should be avoided in professional development
- Alert signs of poor design decisions


WHY LEARN ANTI-PATTERNS?
================================================================================

1. RECOGNIZE THEM IN CODE
   - Identify problematic patterns
   - Know when design is degrading
   - Spot technical debt
   - Flag issues early

2. AVOID THEM
   - Don't repeat mistakes
   - Prevent code quality degradation
   - Save time and effort
   - Better software architecture

3. REFACTOR EXISTING CODE
   - Identify anti-patterns in legacy code
   - Prioritize refactoring
   - Know what to fix first
   - Better maintenance


ANTI-PATTERNS IN THIS DIRECTORY
================================================================================

1. GOD OBJECT (god_object_antipattern.txt)
   ─────────────────────────────────────
   Problem: One massive class doing everything

   Characteristics:
   - 200+ methods
   - 50+ attributes
   - Handles all responsibilities
   - Hard to test and maintain

   Solution:
   - Break into smaller, focused classes
   - Single Responsibility Principle
   - Extract related functionality
   - Create service classes

   Real-world example:
   - UserManagementSystem handling users, payments, emails, logging


2. SPAGHETTI CODE (spaghetti_code_antipattern.txt)
   ────────────────────────────────────────────────
   Problem: Tangled, complex control flow

   Characteristics:
   - Deep nesting (5+ levels)
   - Unclear execution paths
   - Complex boolean logic
   - Goto statements everywhere

   Solution:
   - Extract to smaller methods
   - Guard clauses for early return
   - Reduce nesting depth (max 2-3)
   - Separate concerns

   Real-world example:
   - Order processing with 10 levels of if/else


3. HARDCODED VALUES (hardcoded_values_antipattern.txt)
   ──────────────────────────────────────────────────
   Problem: Values embedded in source code

   Characteristics:
   - Database URLs hardcoded
   - Passwords in source
   - Magic numbers (3.14, 1000, 256)
   - File paths hardcoded
   - API keys in code

   Solution:
   - Use configuration files
   - Use environment variables
   - Use named constants
   - Use dependency injection

   Real-world example:
   - SMTP server hardcoded, DB connection hardcoded


4. GOLD PLATING (gold_plating_antipattern.txt)
   ──────────────────────────────────────────
   Problem: Adding unnecessary features nobody asked for

   Characteristics:
   - Extra features not in requirements
   - Over-engineering
   - Adds complexity
   - Wastes time and resources

   Solution:
   - Define MVP (Minimum Viable Product)
   - Say "no" to extra features
   - Scope management
   - Iterative development (get feedback first)

   Real-world example:
   - Simple email form becomes feature-rich system nobody uses


5. DRY PRINCIPLE VIOLATION (dry_principle_violation.txt)
   ──────────────────────────────────────────────────────
   Problem: Repeating code instead of extracting to reusable pieces

   Characteristics:
   - Copy-pasted code everywhere
   - Same logic in 5 places
   - Magic values duplicated
   - Hard to maintain

   Solution:
   - Extract to common function
   - Use constants for magic numbers
   - Use configuration
   - Apply DRY principle systematically

   Real-world example:
   - Email validation logic copied in 3 different classes


6. CONSTRUCTOR OVERLOADING (constructor_overloading_antipattern.txt)
   ───────────────────────────────────────────────────────────────
   Problem: Too many constructors with different parameter combinations

   Characteristics:
   - 10+ constructors
   - Overlapping parameters
   - Confusing which to use
   - Hard to add new fields

   Solution:
   - Keep 2-3 constructors maximum
   - Use setters for optional parameters
   - Use Builder Pattern for complex initialization
   - Use Factory Methods

   Real-world example:
   - User class with 10 different constructors for different parameter combinations


7. OVERUSE OF GETTER/SETTER (overuse_getter_setter_antipattern.txt)
   ──────────────────────────────────── ───────────────────────────
   Problem: Exposing every field via getter/setter

   Characteristics:
   - No encapsulation
   - Object is just data container
   - No validation
   - No behavioral logic

   Solution:
   - Expose behaviors, not data
   - Private setters
   - Return copies, not references
   - Validate in setters

   Real-world example:
   - BankAccount with setBalance() allowing negative amounts


8. PREMATURE OPTIMIZATION (premature_optimization_antipattern.txt)
   ────────────────────────────────────────────────────────────
   Problem: Optimizing before proving bottlenecks exist

   Characteristics:
   - Complex code without performance gain
   - Sacrificing clarity
   - Optimizing wrong things
   - Not measuring performance

   Solution:
   - Write clear code first
   - Measure before optimizing
   - Profile to find bottlenecks
   - Optimize data-driven

   Real-world example:
   - Writing inline assembly instead of simple addition


9. INHERITANCE OVERUSE (inheritance_overuse_antipattern.txt)
   ──────────────────────────────────────────────────────
   Problem: Deep inheritance hierarchies or inheritance for code reuse

   Characteristics:
   - Deep hierarchies (5+ levels)
   - Inheritance for reuse (wrong!)
   - Violates Liskov Substitution
   - Tight coupling between parent/child

   Solution:
   - Keep hierarchies shallow (max 2-3 levels)
   - Use composition over inheritance
   - Only inherit for IS-A relationships
   - Use interfaces

   Real-world example:
   - Square inheriting from Rectangle (breaks Liskov!)


10. NULL OBJECT PATTERN (null_object_pattern.txt)
    ──────────────────────────────────────────
    NOTE: This is actually a DESIGN pattern, not an anti-pattern
    (Included per request)

    Solution: Providing object as surrogate for null references

    Characteristics:
    - Object that implements interface
    - Provides no-op behavior
    - Eliminates null checks
    - Safe default behavior

    Benefits:
    - No null checks needed
    - Cleaner code
    - Safer than null references
    - Graceful degradation

    Real-world example:
    - NullLogger that does nothing instead of checking for null


ANTI-PATTERNS RELATIONSHIP
================================================================================

STRUCTURAL ISSUES:
├── God Object (too big)
├── Inheritance Overuse (wrong structure)
├── Constructor Overloading (confusing construction)
└── Overuse of Getter/Setter (treats object as data)

CODE QUALITY ISSUES:
├── Spaghetti Code (tangled logic)
├── DRY Principle Violation (repeated code)
├── Gold Plating (unnecessary complexity)
└── Premature Optimization (wrong priorities)

CONFIGURATION ISSUES:
└── Hardcoded Values (inflexible design)


DETECTION CHECKLIST
================================================================================

Is your code suffering from God Object if:
☐ Class has 200+ methods
☐ Class has 50+ attributes
☐ Multiple reasons to change
☐ Hard to understand
☐ Hard to test

Is your code spaghetti code if:
☐ More than 5 levels of nesting
☐ Complex boolean logic
☐ Long methods (100+ lines)
☐ Unclear execution paths

Is your code hardcoded if:
☐ Database URLs in source
☐ Passwords in code
☐ File paths hardcoded
☐ Magic numbers everywhere
☐ Different builds for each environment

Is your code gold plating if:
☐ Features nobody asked for
☐ Features nobody uses
☐ Over-engineered
☐ Delayed release

Is your code DRY violation if:
☐ Same code in 2+ places
☐ Magic values duplicated
☐ Similar logic patterns
☐ Copy-paste detected

Is your code constructor overloading if:
☐ More than 3-4 constructors
☐ Similar parameter patterns
☐ Confusing which to use
☐ Hard to add new fields

Is your code overusing getter/setter if:
☐ Every field has public getter
☐ Every field has public setter
☐ No validation in setters
☐ Object is just data container

Is your code prematurely optimized if:
☐ Complex code without measurement
☐ Sacrificed clarity for speed
☐ No profiling data
☐ Optimized unproven bottlenecks

Is your code overusing inheritance if:
☐ Hierarchy is 4+ levels deep
☐ Using for code reuse (wrong!)
☐ Violates Liskov Substitution
☐ Subclass doesn't IS-A parent


REFACTORING PRIORITY
================================================================================

High Priority (Fix first):
1. Spaghetti Code - affects readability
2. God Object - affects maintainability
3. DRY Violations - affects efficiency

Medium Priority (Fix when possible):
4. Hardcoded Values - affects flexibility
5. Inheritance Overuse - affects extensibility
6. Constructor Overloading - affects usability

Low Priority (Avoid for new code):
7. Overuse of Getter/Setter - affects design
8. Gold Plating - affects efficiency
9. Premature Optimization - affects clarity


METRICS FOR DETECTING ANTI-PATTERNS
================================================================================

Class Size:
- Ideal: < 200 lines of code
- Warning: 200-500 lines
- Anti-pattern: > 500 lines

Method Complexity (Cyclomatic):
- Ideal: < 10
- Warning: 10-20
- Anti-pattern: > 20

Method Length:
- Ideal: < 50 lines
- Warning: 50-100 lines
- Anti-pattern: > 100 lines

Inheritance Depth:
- Ideal: 2-3 levels
- Warning: 4-5 levels
- Anti-pattern: > 5 levels

Number of Constructors:
- Ideal: 1-2
- Warning: 3-4
- Anti-pattern: > 4


CODE REVIEW TIPS
================================================================================

When reviewing code, look for:

1. Classes that are too large
2. Methods that are too long
3. Deep nesting levels
4. Repeated code blocks
5. Multiple constructor signatures
6. Every field having getter/setter
7. Hardcoded values
8. Complex boolean logic
9. No clear responsibility
10. Features that seem unnecessary


BEST PRACTICES TO AVOID ANTI-PATTERNS
================================================================================

1. FOLLOW SOLID PRINCIPLES
   - Single Responsibility
   - Open/Closed
   - Liskov Substitution
   - Interface Segregation
   - Dependency Inversion

2. KEEP CODE SIMPLE
   - KISS (Keep It Simple, Stupid)
   - Avoid over-engineering
   - Clarity over cleverness

3. REFACTOR REGULARLY
   - Extract methods
   - Extract classes
   - Remove duplication
   - Improve names

4. USE DESIGN PATTERNS (Proper ones!)
   - Builder for complex objects
   - Factory for creation
   - Strategy for behavior
   - Composite for structures

5. CODE REVIEWS
   - Peer review catches issues
   - Fresh eyes spot problems
   - Knowledge sharing
   - Prevent anti-patterns

6. MEASURE & PROFILE
   - Don't guess
   - Use profilers
   - Data-driven decisions
   - Optimize what matters

7. WRITE TESTS
   - Easier to test = better design
   - Tests guide design
   - Refactoring safety
   - Prevents regressions


KEY TAKEAWAYS
================================================================================

1. Anti-patterns are common but bad
2. Learn them to recognize and avoid
3. Refactor existing code to remove them
4. Code reviews catch anti-patterns early
5. Follow SOLID principles
6. Keep code simple and clear
7. Refactor regularly
8. Measure before optimizing
9. Use composition over inheritance
10. Extract and reuse code systematically

Anti-patterns summarized:
- God Object: Too many responsibilities
- Spaghetti Code: Tangled logic
- Hardcoded Values: Inflexible design
- Gold Plating: Unnecessary features
- DRY Violations: Repeated code
- Constructor Overloading: Confusing creation
- Getter/Setter Overuse: Broken encapsulation
- Premature Optimization: Wrong priorities
- Inheritance Overuse: Wrong composition
- Null Object: (Actually a design pattern!)
