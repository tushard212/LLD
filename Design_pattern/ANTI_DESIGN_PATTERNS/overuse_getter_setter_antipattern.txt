================================================================================
            OVERUSE OF GETTER/SETTER ANTI-PATTERN
================================================================================

WHAT IS OVERUSE OF GETTER/SETTER?
================================================================================
Overuse of getter/setter occurs when:
- Every field has public getter/setter
- Treats objects like data structures (not objects)
- Violates encapsulation
- No behavioral logic
- Object is just a container
- Code doesn't respect object design

GOOD vs BAD APPROACH:

GOOD:
- Object has behavior and purpose
- Data hidden inside
- Only expose necessary operations
- Internal state protected

BAD:
- Everything public via getter/setter
- Object just holds data
- No encapsulation
- Data exposed to modification


EXAMPLES
================================================================================

// BAD: Over-using Getter/Setter
class BankAccount {
private:
    double balance;
    string accountType;
    vector<Transaction> transactions;
    DateTime lastModified;
    double overdraftLimit;

public:
    // Every field has getter AND setter!
    double getBalance() const { return balance; }
    void setBalance(double b) { balance = b; }  // DANGEROUS!

    string getAccountType() const { return accountType; }
    void setAccountType(const string& t) { accountType = t; }

    vector<Transaction> getTransactions() const { return transactions; }
    void setTransactions(const vector<Transaction>& t) { transactions = t; }

    DateTime getLastModified() const { return lastModified; }
    void setLastModified(const DateTime& d) { lastModified = d; }

    double getOverdraftLimit() const { return overdraftLimit; }
    void setOverdraftLimit(double l) { overdraftLimit = l; }
};

// USAGE - DANGEROUS!
int main() {
    BankAccount account;
    account.setBalance(100);       // Why? Account manages balance!
    account.setBalance(-5000);     // NEGATIVE?! No protection!
    account.setBalance(1000000);   // Arbitrarily change balance? Fraud!

    // Object is just data container
    // No real behavior
    // No protection
}

// PROBLEMS:
// - Can set balance to anything!
// - No validation
// - No business logic
// - Encapsulation broken
// - Data structure, not object


// GOOD: Proper encapsulation with behavior
class BankAccount {
private:
    double balance;
    string accountType;
    vector<Transaction> transactions;
    DateTime lastModified;
    double overdraftLimit;

    // Private helpers
    bool canWithdraw(double amount) const {
        return amount <= (balance + overdraftLimit);
    }

public:
    // Only expose operations (behaviors), not raw data
    void deposit(double amount) {
        if (amount <= 0) throw InvalidAmountException();
        balance += amount;
        transactions.push_back(Transaction("DEPOSIT", amount, DateTime::now()));
        lastModified = DateTime::now();
    }

    bool withdraw(double amount) {
        if (amount <= 0) throw InvalidAmountException();
        if (!canWithdraw(amount)) {
            return false;  // Insufficient funds
        }
        balance -= amount;
        transactions.push_back(Transaction("WITHDRAWAL", amount, DateTime::now()));
        lastModified = DateTime::now();
        return true;
    }

    void transfer(const string& toAccount, double amount) {
        if (!withdraw(amount)) {
            throw InsufficientFundsException();
        }
        // Transfer logic...
        transactions.push_back(Transaction("TRANSFER", amount, DateTime::now()));
    }

    // GETTER ONLY - can't modify
    double getBalance() const { return balance; }

    // GETTER ONLY - transaction history cannot be modified
    vector<Transaction> getTransactionHistory() const {
        return transactions;  // Return copy, not reference
    }

    // Query method - not raw getter
    int getTransactionCount() const {
        return transactions.size();
    }
};

// USAGE - SAFE!
int main() {
    BankAccount account;
    account.deposit(1000);

    if (account.withdraw(100)) {
        cout << "Withdrawal successful\n";
    } else {
        cout << "Insufficient funds\n";
    }

    // Cannot set balance arbitrarily
    // account.setBalance(-5000);  // ERROR: No setter!

    // Only operations available
    // Balance is protected
}


PROBLEMS WITH GETTER/SETTER OVERUSE
================================================================================

1. BROKEN ENCAPSULATION
   - Data exposed
   - No protection
   - Anyone can modify state
   - No validation

2. NO REAL OBJECT
   - Just data container
   - No behavior
   - Should be struct, not class
   - Object-oriented design failed

3. HARD TO MAINTAIN
   - Change data structure = change all callers
   - Refactoring nightmare
   - Tight coupling
   - Can't hide internal details

4. DIFFICULT TO VALIDATE
   - No validation when setting
   - Invalid state possible
   - Bugs from invalid state
   - Hard to debug

5. DIFFICULT TO TRACK CHANGES
   - Data can change anywhere
   - Hard to trace modifications
   - Unknown who changed what
   - Audit trail impossible

6. THREAD SAFETY ISSUES
   - Data can change from any thread
   - Race conditions
   - Data corruption possible
   - Synchronization nightmare

7. VIOLATES TELL DON'T ASK PRINCIPLE
   - Caller asks for data
   - Caller operates on data
   - Object just holds data
   - Should be object making decisions

8. PERFORMANCE ISSUES
   - Returning vectors = copies
   - Unnecessary memory allocation
   - Performance degradation
   - Memory waste


HOW TO FIX GETTER/SETTER OVERUSE
================================================================================

1. EXPOSE BEHAVIORS, NOT DATA
   - Ask "what can this object do?"
   - Not "what data does it store?"
   - deposit(), withdraw(), not setBalance()
   - validate(), process(), not setStatus()

2. MAKE SETTERS PRIVATE
   - Initialization only in constructor
   - Internal use only
   - External: use methods
   - Protects invariants

3. RETURN COPIES, NOT REFERENCES
   - Don't return internal collections
   - Collection can't be modified externally
   - Protects from modification
   - Encapsulation maintained

4. ADD VALIDATION
   - Setters check values
   - Reject invalid state
   - Throw exceptions
   - Maintain invariants

5. USE IMMUTABILITY
   - Don't allow modification
   - Create new objects instead
   - Functional style
   - Thread-safe

6. TELL DON'T ASK
   - account.withdraw(100)
   - Not: account.setBalance(account.getBalance() - 100)
   - Object does the work
   - Caller just tells what to do

7. HIDE INTERNAL COLLECTIONS
   - Don't expose lists/vectors directly
   - Return copies or iterators
   - Return immutable views
   - Caller can't modify internals

8. USE QUERY METHODS
   - isValid(), canWithdraw()
   - Not getStatus(), getBalance()
   - Intent is clearer
   - Less intrusive


WHEN GETTERS/SETTERS ARE APPROPRIATE
================================================================================

✓ Use getters/setters for:
- Simple data transfer objects (DTOs)
- Value objects that are immutable
- Configuration objects
- When you need computed properties

✗ Don't use getters/setters for:
- Core domain objects
- Objects with complex behavior
- Objects that enforce invariants
- Objects that own other objects


GOOD PATTERNS
================================================================================

1. ENCAPSULATION PATTERN
   - Private data
   - Public methods (behaviors)
   - Validation in methods
   - Invariants maintained

2. DATA TRANSFER OBJECT (DTO)
   - Simple data holder
   - Getters/setters OK
   - No business logic
   - Just transfers data

3. VALUE OBJECT
   - Immutable
   - No setters
   - Behaviors only
   - Thread-safe

4. BUILDER PATTERN
   - Complex initialization
   - Single constructor
   - Fluent setters (immutable)
   - Immutable result


KEY DIFFERENCES
================================================================================

OBJECTS (With behavior):
```
class BankAccount {
private:
    double balance;

public:
    void deposit(double amount) { balance += amount; }
    void withdraw(double amount) { balance -= amount; }
    double getBalance() const { return balance; }  // Getter only
}
```

DATA OBJECTS (No behavior):
```
struct PersonData {
    string name;
    string email;
    int age;
};

// Use struct, not class
// Or make everything public
// Or use proper encapsulation with methods
```


ADVANTAGES
================================================================================

(Why developers use getters/setters)

1. CONVENIENT
   - Easy to access data
   - Seems user-friendly

2. FLEXIBILITY
   - Can change implementation
   - Still expose same data


DISADVANTAGES
================================================================================

(Why overuse is BAD)

1. BROKEN ENCAPSULATION
   - Data exposed
   - No protection
   - Can't maintain invariants

2. NOT REAL OBJECTS
   - No behavior
   - Just data containers
   - Should use struct

3. HARD TO MAINTAIN
   - Changes ripple everywhere
   - Tight coupling
   - Can't hide details

4. NO VALIDATION
   - Invalid state possible
   - Bugs from bad data
   - Hard to debug

5. THREAD SAFETY
   - Race conditions
   - Data corruption
   - Synchronization nightmare


KEY TAKEAWAYS
================================================================================

1. Expose operations, not data
2. Keep data private
3. Use methods with validation
4. Follow "Tell Don't Ask" principle
5. Use immutability when possible
6. Only getters for public data
7. Setters are initialization only
8. Objects should have behavior
9. Encapsulation is critical
10. Data objects ≠ real objects
