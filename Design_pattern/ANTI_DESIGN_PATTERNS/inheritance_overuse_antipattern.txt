================================================================================
                   INHERITANCE OVERUSE ANTI-PATTERN
================================================================================

WHAT IS INHERITANCE OVERUSE?
================================================================================
Inheritance overuse occurs when:
- Using inheritance for code reuse (wrong!)
- Creating deep inheritance hierarchies (5+ levels)
- Inherited classes don't fit "is-a" relationship
- Inheritance used instead of composition
- Violates Liskov Substitution Principle
- Makes class hierarchies confusing

Key principle: "Composition over inheritance"


PROBLEMS WITH INHERITANCE
================================================================================

1. TIGHT COUPLING
   - Parent and child tightly coupled
   - Change parent = affects all children
   - Hard to test independently
   - Fragile base class problem

2. FRAGILE HIERARCHY
   - Small change ripples down
   - Subclass depends on parent implementation
   - Unexpected behavior changes
   - Hard to maintain

3. HARD TO UNDERSTAND
   - Must understand entire hierarchy
   - Where does behavior come from?
   - Multiple levels to trace
   - Mental model is complex

4. LIMITED FLEXIBILITY
   - Can't change parent without affecting children
   - Can't isolate behaviors
   - Adding new behaviors requires new subclass
   - Combinatorial explosion

5. BREAKS LISKOV SUBSTITUTION
   - Subclass doesn't properly substitute parent
   - Unexpected behavior
   - Violates contracts
   - Violates SOLID principles


EXAMPLES OF INHERITANCE OVERUSE
================================================================================

Example 1: Over-deep inheritance hierarchy

// BAD: Deep inheritance hierarchy
class Shape { }

class TwoDShape : public Shape { }

class Polygon : public TwoDShape { }

 class RegularPolygon : public Polygon { }

class Quadrilateral : public RegularPolygon { }

class Rectangle : public Quadrilateral { }

class Square : public Rectangle { }

// 7 levels of inheritance!
// Must understand all levels
// Complex to add new shape
// Hard to modify
// COUNTERPRODUCTIVE


// GOOD: Shallow, appropriate hierarchy
class Shape { };

class Rectangle : public Shape { };

class Square : public Shape { };

class Circle : public Shape { };

// 2 levels only
// Clear relationships
// Easy to add new shapes
// Simple to understand


Example 2: Inheritance for code reuse (WRONG)

// BAD: Using inheritance for code reuse
class DataProcessor {
protected:
    string fileName;
    vector<string> data;

    void loadFile() {
        // File loading logic
    }

public:
    virtual void process() = 0;
};

class CSVProcessor : public DataProcessor {
public:
    void process() override {
        loadFile();  // From parent
        // Parse CSV
    }
};

class JSONProcessor : public DataProcessor {
public:
    void process() override {
        loadFile();  // From parent (but JSON loading different!)
        // Parse JSON
    }
};

// PROBLEMS:
// - loadFile() doesn't make sense for JSON
// - Inheritance forced for code reuse
// - Not true "is-a" relationship
// - Breaks Liskov Substitution
// - Confusing hierarchy


// GOOD: Use composition
class FileLoader {
public:
    vector<string> load(const string& fileName) {
        // Generic file loading
    }
};

class CSVParser {
private:
    FileLoader loader;

public:
    void process(const string& fileName) {
        vector<string> data = loader.load(fileName);
        // Parse CSV
    }
};

class JSONParser {
private:
    FileLoader loader;

public:
    void process(const string& fileName) {
        vector<string> data = loader.load(fileName);
        // Parse JSON
    }
};

// BENEFITS:
// - Clear separation of concerns
// - Each class has one job
// - No confusing hierarchy
// - Reusable components
// - Testable


Example 3: Wrong "is-a" relationship

// BAD: Wrong inheritance relationship
class Rectangle {
private:
    int width, height;

public:
    void setWidth(int w) { width = w; }
    void setHeight(int h) { height = h; }
    int getArea() { return width * height; }
};

class Square : public Rectangle {
    // WRONG: Square is NOT a Rectangle!
    // Square has constraints
    // Can't setWidth without setHeight
    // Breaks Liskov Substitution Principle

public:
    void setWidth(int w) override {
        setWidth(w);
        setHeight(w);  // Constraint
    }

    void setHeight(int h) override {
        setWidth(h);
        setHeight(h);  // Constraint
    }
};

// PROBLEM:
Rectangle* shape = new Square();
shape->setWidth(5);
shape->setHeight(10);
// Square is now  NOT a square! (5x10)
// Contract violated!


// GOOD: Use composition or correct hierarchy
class Shape {
public:
    virtual int getArea() = 0;
};

class RectangleShape : public Shape {
private:
    int width, height;

public:
    RectangleShape(int w, int h) : width(w), height(h) { }
    int getArea() override { return width * height; }
};

class SquareShape : public Shape {
private:
    int side;

public:
    SquareShape(int s) : side(s) { }
    int getArea() override { return side * side; }
};

// NO inheritance problem
// Each class clear
// No constraint violation
// Liskov principle satisfied


WHEN TO USE INHERITANCE
================================================================================

✓ Use inheritance when: TRUE "IS-A" relationship exists
- Dog is a Animal
- Circle is a Shape
- Truck is a Vehicle

✓ Use when: Subclass can substitute parent (Liskov Substitution)
- Anywhere Animal used, Dog works
- Everywhere Shape used, Circle works

✗ Don't use when: Just want to reuse code
- Use composition instead
- Share behavior differently
- Encapsulate common code

✗ Don't use when: "IS-A" doesn't make sense
- Square is not a Rectangle
- Penguin is not a Bird (can't fly!)
- Duck doesn't "is-a" Swimmer


COMPOSITION ALTERNATIVE
================================================================================

// Pattern: Composition instead of inheritance
class Car {
private:
    Engine engine;    // HAS-A
    Transmission tx;  // HAS-A
    Wheels wheels;    // HAS-A

public:
    void drive() {
        engine.start();
        tx.shift();
        wheels.rotate();
    }
};

// BENEFITS:
// - Clear relationships
// - Easy to understand
// - Easy to test
// - Flexible combinations
// - No hierarchy headaches


CHOOSING: INHERITANCE vs COMPOSITION
================================================================================

Use inheritance when:
- True IS-A relationship
- Polymorphic behavior needed
- Shared interface needed
- Liskov Substitution works

Use composition when:
- HAS-A relationship
- Reusing code/behavior
- Flexibility needed
- Avoiding hierarchy
- Multiple capabilities


ADVANTAGES
================================================================================

(Why developers overuse inheritance)

1. CODE REUSE
   - Can share code from parent
   - Seems efficient
   - Avoids duplication

2. POLYMORPHISM
   - Treat subclasses uniformly
   - Useful for extensibility
   - Natural in OOP


DISADVANTAGES
================================================================================

(Why overuse is BAD)

1. TIGHT COUPLING
   - Parent and child coupled
   - Changes ripple
   - Hard to change

2. FRAGILE
   - Small changes break subclass
   - Unexpected behavior
   - Hard to maintain

3. HIERARCHY COMPLEXITY
   - Deep trees hard to understand
   - Multiple levels to trace
   - Mental model complex

4. LIMITED FLEXIBILITY
   - Can't mix behaviors
   - Can't break inheritance chain
   - Rigid structure

5. LSP VIOLATIONS
   - Subclass may not substitute
   - Unexpected behavior
- Violates SOLID


HOW TO FIX INHERITANCE OVERUSE
================================================================================

1. KEEP HIERARCHIES SHALLOW
   - Maximum 2-3 levels
   - Easy to understand
   - Easy to maintain
   - Clear relationships

2. USE COMPOSITION
   - HAS-A instead of IS-A
   - More flexible
   - Clearer design
   - Easier to test

3. CHECK LISKOV SUBSTITUTION
   - Can subclass actually substitute?
   - Doesn't violate contracts
   - Behavior consistent
   - No surprises

4. ASK "IS-A OR HAS-A?"
   - Penguin is Bird? (No - can't fly!)
   - Car has Engine? (Yes!)
- Clear thinking

5. PREFER INTERFACES
   - Define contracts
   - Multiple interfaces
   - Flexible combinations
   - Clear intent

6. USE MIXINS / TRAITS
   - Multiple behavior sources
   - No deep hierarchy
   - Flexible combinations
   - Modern alternative


KEY TAKEAWAYS
================================================================================

1. Inheritance for IS-A relationships only
2. Composition for HAS-A relationships
3. Composition over inheritance
4. Keep hierarchies shallow
5. Check Liskov Substitution Principle
6. Deep hierarchies are red flag
7. Code reuse via composition, not inheritance
8. Don't use inheritance for convenience
9. Prefer interfaces to base classes
10. Flexibility more important than reuse
