================================================================================
                        BRIDGE DESIGN PATTERN
================================================================================

1. WHAT IS BRIDGE DESIGN PATTERN?
================================================================================
The Bridge Design Pattern decouples an abstraction from its implementation so
the two can vary independently. It's about separating the interface from the
implementation using composition instead of inheritance.

Think of it like a real-world scenario:
  - Car type (Sedan, SUV, Sports Car) - the abstraction
  - Engine type (Petrol, Diesel, Electric) - the implementation
  - A Sedan can have Petrol OR Diesel OR Electric engine
  - A Sports Car can have Petrol OR Diesel OR Electric engine
  - Without Bridge: Create 9 classes (Sedan×3 + SUV×3 + Sports×3)
  - With Bridge: Create 3 car types + 3 engines = 6 classes, infinite combinations

Components:
- Abstraction: High-level interface (Car)
- RefinedAbstraction: Extended abstraction (Sedan, SUV, SportsCar)
- Implementor: Implementation interface (Engine)
- ConcreteImplementor: Actual implementation (PetrolEngine, DieselEngine, ElectricEngine)
- Bridge: Composition link between abstraction and implementation

================================================================================
2. WHY WAS IT NEEDED?
================================================================================
Problems it solves:

1. COMBINATORIAL EXPLOSION WITH INHERITANCE
   - Two dimensions of variation: car type AND engine type
   - With inheritance: 3 cars × 3 engines = 9 classes!
   - If add 4th car type: need 12 classes
   - If add 4th engine: need 16 classes
   - Explosion grows exponentially

2. TIGHT COUPLING BETWEEN ABSTRACTION AND IMPLEMENTATION
   - Changing engine implementation affects car classes
   - Can't change engine without recompiling car classes
   - Violates dependency inversion principle

3. SHARING IMPLEMENTATION ACROSS ABSTRACTIONS
   - Multiple abstractions need same implementation
   - Can't reuse - forced to duplicate code
   - Example: Both Sedan and SUV might want DieselEngine behavior

4. COMPILE-TIME BINDING
   - With inheritance, decision made at compile time
   - Can't switch implementations at runtime
   - No flexibility

Example Problem (without Bridge - Explosion):
```
// Without Bridge - Combinatorial explosion!

class Sedan { };
class SedanPetrol : public Sedan { };  // Class 1
class SedanDiesel : public Sedan { };  // Class 2
class SedanElectric : public Sedan { }; // Class 3

class SUV { };
class SUVPetrol : public SUV { };      // Class 4
class SUVDiesel : public SUV { };      // Class 5
class SUVElectric : public SUV { };    // Class 6

class SportsCar { };
class SportsCarPetrol : public SportsCar { }; // Class 7
class SportsCarDiesel : public SportsCar { }; // Class 8
class SportsCarElectric : public SportsCar { }; // Class 9

// Total: 9 classes just for 3 cars × 3 engines!
// If we add one more car: +3 classes = 12 total
// If we add one more engine: +3 classes = 12 total
// Exponential growth!

// Also, code duplication:
// SedanDiesel and SUVDiesel both have diesel logic - can't share!
```

With Bridge:
```
// With Bridge - Clean separation!

// Abstraction side
abstract class Car {
    Engine* engine;  // Composition instead of inheritance!
};

class Sedan : public Car { };
class SUV : public Car { };
class SportsCar : public Car { };

// Implementation side
abstract class Engine {
    virtual void start() = 0;
};

class PetrolEngine : public Engine { };
class DieselEngine : public Engine { };
class ElectricEngine : public Engine { };

// Total: 3 cars + 3 engines = 6 classes
// Any car can have any engine!
// DieselEngine reused by all cars!
// Add 4th car: just 1 new class (not 3)
```

================================================================================
3. ADVANTAGES AND DISADVANTAGES
================================================================================

ADVANTAGES:
===========
✓ No Combinatorial Explosion: N abstractions + M implementations = N+M classes
  - Not N×M classes
  - Scales linearly instead of exponentially
  - Easy to add new car types or engine types

✓ Decoupling: Abstraction independent from implementation
  - Car doesn't know concrete engine type
  - Engine doesn't know concrete car type
  - Can change each independently
  - Follows Open/Closed Principle

✓ Runtime Flexibility: Switch implementations at runtime
  - Create Sedan with PetrolEngine
  - Later, switch to ElectricEngine
  - No recompilation needed
  - Dynamic binding possible

✓ Code Reuse: Share implementation across abstractions
  - DieselEngine used by Sedan, SUV, SportsCar
  - Not duplicated in each car class
  - Single place to fix diesel bugs
  - DRY principle

✓ Separation of Concerns: Abstraction and implementation separate
  - Car developer doesn't touch engine code
  - Engine developer doesn't touch car code
  - Different teams can work independently
  - Easier to understand each part

✓ Easier Testing: Can mock implementations
  - Test Sedan with MockEngine
  - Test PetrolEngine with MockCar
  - Independent unit testing

✓ Easier Modification: Change implementation without affecting abstraction
  - Improve PetrolEngine
  - All cars using it benefit
  - Sedan code unchanged

DISADVANTAGES:
==============
✗ Increased Complexity: More classes and abstraction
  - 6 classes instead of 9, but more complex
  - Harder to understand for beginners
  - More indirection to follow

✗ Performance Overhead: Indirection through interface
  - Extra dereference through bridge (engine pointer)
  - Method calls go through virtual function table
  - Might matter in performance-critical code

✗ Over-engineering: For simple cases
  - If only one variation dimension, Bridge is overkill
  - Composition adds complexity for no benefit
  - YAGNI (You Aren't Gonna Need It)

✗ Not Obvious Relationship: Harder to understand design
  - Composition is less obvious than inheritance
  - Someone reading code might not understand the design
  - Need documentation

✗ Null Checks Needed: Bridge might be null
  - Must check if implementation exists
  - Error handling complexity
  - Potential for null pointer exceptions

✗ Testing Complexity: Multiple levels to test
  - Need to test abstraction
  - Need to test implementation
  - Need to test their interaction

✗ Constrained by Implementation Interface: All abstractions must use same interface
  - Can't add car-specific engine methods
  - Engine interface must be generic enough
  - Might be problematic if implementations very different

✗ Hidden Dependencies: Dependency injection complexity
  - Where are implementations created?
  - Who manages their lifecycle?
  - Dependency injection adds infrastructure

4. BRIDGE VS ADAPTER VS STRATEGY
================================================================================

BRIDGE vs ADAPTER:
==================
Bridge:
  - Used at design time
  - Separates abstraction from implementation
  - Planned in advance
  - Decoupling is goal

Adapter:
  - Used when interfaces don't match
  - Afterthought to fix incompatibility
  - Not planned in advance
  - Interface conversion is goal

BRIDGE vs STRATEGY:
===================
Bridge:
  - Separates abstraction from implementation
  - Permanent structure
  - Both abstraction and implementation can vary
  - Inheritance + Composition

Strategy:
  - Encapsulates algorithm
  - Runtime selection of behavior
  - Only behavior varies
  - Context uses strategy

BRIDGE vs TEMPLATE METHOD:
==========================
Bridge:
  - Composition-based
  - Runtime flexibility
  - Both sides can extend
  - Decoupling is goal

Template Method:
  - Inheritance-based
  - Compile-time structure
  - Only implementation side varies
  - Code reuse is goal

5. CLASS FLOW / ARCHITECTURE
================================================================================

PROBLEM: Combinatorial Explosion
=================================
                    Car
                  /  |  \
            Sedan SUV SportsX
            / | \ / | \  / | \
           S  S  S S S S  S S S
           P  D  E P D E  P D E
                (9 classes!)

Each combination is a class!
Explosion as variations grow!


SOLUTION: Bridge Pattern
========================

        ┌──────────────────────────────────────┐
        │    <<Abstraction>>                   │
        │        Car                           │
        ├──────────────────────────────────────┤
        │ # engine: Engine*                    │
        │                                      │
        ├──────────────────────────────────────┤
        │ + getType()                          │
        │ + start()          ◄─ Delegates    │
        │ + drive()          ◄─ to engine    │
        │ + stop()           ◄─ implementation
        └──────────────┬──────────────────────┘
                       │Uses
                       │(composition)
                       │
        ┌──────────────┼──────────────┬──────────────┐
        │              │              │              │
   ┌──────────────┐ ┌──────────┐ ┌──────────────┐
   │   Sedan      │ │   SUV    │ │ SportsCar    │
   │              │ │          │ │              │
   │ + getType()  │ │getType() │ │ + getType()  │
   └──────────────┘ └──────────┘ └──────────────┘
   (Refined        (Refined     (Refined
    Abstraction)    Abstraction) Abstraction)

                BRIDGE (composition link)
                ↓ ↓ ↓
        ┌────────────────────────────────┐
        │  <<Implementor>>               │
        │      Engine                    │
        ├────────────────────────────────┤
        │ + start(): void = 0            │
        │ + accelerate(): void = 0       │
        │ + stop(): void = 0             │
        └──────────────┬─────────────────┘
                       △
        ┌──────────────┼──────────────┬──────────────┐
        │              │              │              │
   ┌──────────────┐ ┌──────────┐ ┌──────────────┐
   │ PetrolEngine │ │Diesel    │ │ Electric     │
   │              │ │Engine    │ │ Engine       │
   │ + start()    │ │+ start() │ │ + start()    │
   │ + accel()    │ │+ accel() │ │ + accel()    │
   └──────────────┘ └──────────┘ └──────────────┘
   (Concrete       (Concrete    (Concrete
    Implementor)    Implementor)  Implementor)

RESULT:
=======
3 Abstractions + 3 Implementations = 6 classes
VERSUS
3 × 3 = 9 classes (without Bridge)

Any Car can use any Engine!
    Sedan + PetrolEngine
    Sedan + DieselEngine
    Sedan + ElectricEngine
    SUV + PetrolEngine
    ... and so on!


Flow:
=====
Client calls: sedan.drive()
         │
         ▼
    ┌──────────────────┐
    │ Sedan.drive()    │
    │ (uses bridge)    │
    └──────────────────┘
         │
         │ delegates to
         ▼
    ┌──────────────────┐
    │ engine.accelerate
    │ ()               │
    └──────────────────┘
         │
         │ polymorphism
         ▼
    ┌──────────────────┐
    │ Actual Engine    │
    │ (Diesel/Petrol/  │
    │  Electric)       │
    └──────────────────┘

6. COMPLETE C++ EXAMPLE: CAR AND ENGINE BRIDGE
================================================================================

#include <iostream>
#include <string>
#include <memory>
using namespace std;

// ============= IMPLEMENTOR: ENGINE INTERFACE =============

abstract class Engine {
public:
    virtual ~Engine() {}

    virtual void start() = 0;

    virtual void accelerate() = 0;

    virtual void brake() = 0;

    virtual void stop() = 0;

    virtual string getEngineType() const = 0;

    virtual int getPower() const = 0;
};

// ============= CONCRETE IMPLEMENTORS: SPECIFIC ENGINES =============

class PetrolEngine : public Engine {
public:
    void start() override {
        cout << "    [PetrolEngine] Starting with petrol ignition..." << endl;
        cout << "    [PetrolEngine] Spark plugs fired, engine roaring!" << endl;
    }

    void accelerate() override {
        cout << "    [PetrolEngine] Accelerating! Fuel injection active..." << endl;
        cout << "    [PetrolEngine] RPM: 3000 → 5000" << endl;
    }

    void brake() override {
        cout << "    [PetrolEngine] Engine braking engaged..." << endl;
        cout << "    [PetrolEngine] RPM: 5000 → 2000" << endl;
    }

    void stop() override {
        cout << "    [PetrolEngine] Engine stopped." << endl;
    }

    string getEngineType() const override {
        return "Petrol Engine (2.0L)";
    }

    int getPower() const override {
        return 150;  // 150 HP
    }
};

class DieselEngine : public Engine {
public:
    void start() override {
        cout << "    [DieselEngine] Starting diesel engine (slow startup)..." << endl;
        cout << "    [DieselEngine] Compression ignition engaged!" << endl;
    }

    void accelerate() override {
        cout << "    [DieselEngine] Accelerating with high torque..." << endl;
        cout << "    [DieselEngine] RPM: 2000 → 4000 (steady acceleration)" << endl;
    }

    void brake() override {
        cout << "    [DieselEngine] Engine braking with high torque..." << endl;
        cout << "    [DieselEngine] RPM: 4000 → 1500" << endl;
    }

    void stop() override {
        cout << "    [DieselEngine] Diesel engine stopped." << endl;
    }

    string getEngineType() const override {
        return "Diesel Engine (3.0L)";
    }

    int getPower() const override {
        return 200;  // 200 HP (more torque)
    }
};

class ElectricEngine : public Engine {
public:
    void start() override {
        cout << "    [ElectricEngine] Silent startup, battery activated..." << endl;
        cout << "    [ElectricEngine] Electric motor ready!" << endl;
    }

    void accelerate() override {
        cout << "    [ElectricEngine] Instant acceleration (electric torque)..." << endl;
        cout << "    [ElectricEngine] RPM: 0 → 8000 (instant response)" << endl;
    }

    void brake() override {
        cout << "    [ElectricEngine] Regenerative braking engaged..." << endl;
        cout << "    [ElectricEngine] Energy recovered to battery" << endl;
    }

    void stop() override {
        cout << "    [ElectricEngine] Electric motor powered off (silent)." << endl;
    }

    string getEngineType() const override {
        return "Electric Engine (300kW)";
    }

    int getPower() const override {
        return 300;  // 300 HP equivalent
    }
};

// ============= ABSTRACTION: CAR INTERFACE =============

abstract class Car {
protected:
    shared_ptr<Engine> engine;  // THE BRIDGE!
    string color;

public:
    Car() : color("Unknown") {}

    virtual ~Car() {}

    // Set engine implementation (bridge)
    void setEngine(shared_ptr<Engine> eng) {
        engine = eng;
    }

    // Common functionality
    void start() {
        cout << getCarType() << " starting..." << endl;
        cout << "Engine: " << engine->getEngineType() << endl;
        engine->start();
        cout << "Car is ready to drive!" << endl;
    }

    void drive() {
        cout << "\n" << getCarType() << " is driving..." << endl;
        engine->accelerate();
    }

    void doEmergencyStop() {
        cout << "\n" << getCarType() << " emergency stop!" << endl;
        engine->brake();
    }

    void stop() {
        cout << "\n" << getCarType() << " stopping..." << endl;
        engine->stop();
    }

    void showSpecs() {
        cout << "\n╔════════════════════════════════════╗" << endl;
        cout << "║  " << getCarType() << endl;
        cout << "╠════════════════════════════════════╣" << endl;
        cout << "║  Color: " << color << endl;
        cout << "║  Engine: " << engine->getEngineType() << endl;
        cout << "║  Power: " << engine->getPower() << " HP" << endl;
        cout << "╚════════════════════════════════════╝" << endl;
    }

    virtual string getCarType() const = 0;
};

// ============= REFINED ABSTRACTIONS: SPECIFIC CARS =============

class Sedan : public Car {
public:
    Sedan(const string& c = "Black") {
        color = c;
    }

    string getCarType() const override {
        return "Sedan";
    }
};

class SUV : public Car {
public:
    SUV(const string& c = "White") {
        color = c;
    }

    string getCarType() const override {
        return "SUV";
    }
};

class SportsCar : public Car {
public:
    SportsCar(const string& c = "Red") {
        color = c;
    }

    string getCarType() const override {
        return "SportsCar";
    }

    void drive() override {
        cout << "\n" << getCarType() << " SPEEDING UP! (high performance mode)" << endl;
        engine->accelerate();
        engine->accelerate();  // Extra acceleration for sports car
    }
};

// ============= CLIENT CODE =============

int main() {
    cout << "===== BRIDGE DESIGN PATTERN EXAMPLE =====" << endl;
    cout << "Car Types with Different Engines\n" << endl;

    // ===--- Scenario 1: Sedan with Petrol Engine ---===
    cout << string(70, '=') << endl;

    cout << "  - 3 Cars + 3 Engines = 6 classes (not 9!)" << endl;
    cout << "  - Any car type can use any engine type" << endl << endl;

    cout << "✓ Runtime Flexibility:" << endl;

    cout << "  - SUV started with Diesel, switched to Electric" << endl << endl;

    cout << "✓ Code Reuse:" << endl;
    cout << "  - PetrolEngine used by Sedan AND SportsCar" << endl;
    cout << "  - No duplication" << endl << endl;

    cout << "✓ Decoupling:" << endl;
    cout << "  - Car doesn't know concrete engine type" << endl;
    cout << "  - Engine doesn't know concrete car type" << endl;
    cout << "  - Independent development possible" << endl << endl;

    cout << "✓ Easy Extension:" << endl;
    cout << "  - Add new car type: just 1 new class" << endl;
    cout << "  - Add new engine: just 1 new class" << endl;
    cout << "  - No modification to existing code" << endl;

    cout << string(70, '=') << "\n" << endl;

    return 0;
}

✓ Database Access (Abstraction vs Implementation)
  - High-level query interface vs low-level SQL implementations
  - Same code works with MySQL, PostgreSQL, SQLite

✓ GUI Frameworks
  - Platform-independent GUI (Abstraction)
  - Windows/Linux/Mac specific implementations
  - Same code on all platforms

================================================================================
8. WHEN TO USE BRIDGE PATTERN
================================================================================

USE WHEN:
✓ Two independent dimension of variation
  - Cars AND engines (both vary independently)
  - Shape AND rendering (both vary independently)
✓ Want to avoid combinatorial explosion
  - N × M combinations without N×M classes
✓ Want to share implementation across abstractions
  - Multiple abstractions use same implementation
✓ Want runtime flexibility
  - Change implementation at runtime
✓ Want to decouple abstraction from implementation
  - They should vary independently
✓ Multiple teams developing
  - One team on abstraction, another on implementation

AVOID WHEN:
✗ Only one dimension of variation
  - Strategy pattern might be simpler
✗ No expected growth
  - Over-engineering
✗ Very simple cases
  - Added complexity not worth it
✗ Performance critical
  - Extra indirection might matter
✗ No clear abstraction-implementation boundary
  - Hard to apply pattern

10. BEST PRACTICES
================================================================================

DO:
✓ Only use when you have two independent dimensions of variation
✓ Clearly separate abstraction from implementation
✓ Keep abstractions stable, implementations can change
✓ Use dependency injection to provide implementations
✓ Document which classes are abstraction vs implementation
✓ Test abstraction independently from implementation
✓ Keep bridge interface small and focused

DON'T:
✗ Use for single dimension of variation (use Strategy instead)
✗ Mix business logic between abstraction and implementation
✗ Create deep hierarchies
✗ Force implementations to adapt to abstractions
✗ Over-engineer if not truly needed
✗ Ignore performance implications of indirection
✗ Leave bridge uninitialized (null pointer risks)

DESIGN CHECKLIST:
□ Are there truly two independent dimensions?
□ Will implementations need to vary independently?
□ Is expected to grow (add more cars or engines)?
□ Is abstraction-implementation boundary clear?
□ Can abstraction use implementations through interface?
□ Do all implementations share core behavior?
□ Is indirection acceptable for problem?

11. ADVANTAGES vs DISADVANTAGES IN ACTION
================================================================================

SCALABILITY ANALYSIS:

Without Bridge (Inheritance Explosion):
  Cars:    Sedan, SUV, SportsCar (3)
  Engines: Petrol, Diesel (2)
  Classes: 3 × 2 = 6 classes

  Cars: Sedan, SUV, SportsCar, Coupe, Van (5)
  Engines: Petrol, Diesel, Electric (3)
  Classes: 5 × 3 = 15 classes

  Exponential growth!


With Bridge (Decoupling):
  Cars: 3 classes (Sedan, SUV, SportsCar)
  Engines: 2 classes (Petrol, Diesel)
  Total: 3 + 2 = 5 classes

  Cars: 5 classes (add Coupe, Van)
  Engines: 3 classes (add Electric)
  Total: 5 + 3 = 8 classes

  Linear growth!


CODE DUPLICATION:

Without Bridge:
  Diesel logic duplicated in:
    - SedanDiesel
    - SUVDiesel
    - SportsDiesel
    - (any new car type with diesel)

  Bug fix in diesel: modify all classes!


With Bridge:
  Diesel logic in: DieselEngine (one place!)
  All car types automatically use fixed diesel logic
  Bug fix in diesel: fix one class!
